2. 扣费优先级逻辑 (RelayRecordService.java)
================================================================================

关键位置: RelayRecordService.java 第 115-126 行

扣费流程:
  API成功且有成本(cost > 0)
    ↓
  判断: 有活跃订阅 && 额度未耗尽?
    ├─ YES → 扣订阅额度 (subscriptionService.deductQuota)
    └─ NO  → 扣用户余额 (userService.deductByAuthUserId)
    ↓
  记录每日用量 (dailyQuotaService.addCost)
    ↓
  落记录 (relay_records 表)

代码:
if (item.getCost() != null && item.getCost().compareTo(BigDecimal.ZERO) > 0) {
    Long groupId = group != null ? group.getId() : null;
    
    if (subscriptionService.hasQuotaRemaining(apiKey.getOwnerId(), groupId)) {
        subscriptionService.deductQuota(apiKey.getOwnerId(), groupId, item.getCost());
    } else {
        userService.deductByAuthUserId(apiKey.getOwnerId(), item.getCost());
    }
    dailyQuotaService.addCost(apiKey.getOwnerId(), item.getCost());
}

5. 成本计算 (CostCalculationService.calculate)
================================================================================

文件: CostCalculationService.java 第 46-64 行

公式:
  成本(¥) = (inputTokens × inputPrice + outputTokens × outputPrice) ÷ 1,000,000 × groupRatio

精度: BigDecimal, 10位小数

示例:
  input=100, output=50, inputPrice=3.0, outputPrice=6.0, groupRatio=1.5
  = (100×3 + 50×6) ÷ 1M × 1.5
  = 600 ÷ 1M × 1.5
  = 0.0009 ¥

10. 已知问题
================================================================================

【问题 1 - 🔴 严重】余额可能被扣成负数
  文件: UserService.java#deductByAuthUserId() 184行
  原因: 无下限检查
  风险: 余额可能为负值

【问题 2 - 🔴 严重】扣费异常被静默吞掉
  文件: RelayRecordService.java#record() 115-126行
  原因: 无 try-catch，异常直接传播但上层不处理
  风险: "有记录无扣费" 账实不符

【问题 3 - 🟠 中等】分组计费类型未实现
  文件: GroupItem.billingType, billingAmount 有定义但无使用
  风险: 包年/企业折扣无法使用

【问题 4 - 🟠 中等】定价缓存无热更新
  文件: CostCalculationService#refreshCache() 仅启动时调用
  风险: 修改定价需重启服务

【问题 5 - 🟡 轻微】每日配额检查位置靠后
  超额请求仍会转发到上游 API
