# OpenClaw 角色拆分部署设计

## 背景与目标
- 目标：在单台服务器上部署 OpenClaw，按角色划分实例，优先保障稳定性与隔离性。
- 规模：中等规模，持续运行、偶尔并发。

## 架构
- 实例划分：
  - PM 实例独立运行，独立配置/日志/运行目录。
  - DEV+QA 实例共享运行，统一配置/日志/运行目录。
- 进程守护：systemd 或 supervisor 分别拉起两个实例，互不影响。
- 指令入口：
  - 推荐多 Bot 隔离：PM 一个 Bot，DEV+QA 一个 Bot，降低误投与权限混乱。
  - 可选单 Bot 路由：命令带角色标签路由（运维更省，但风险更高）。

## 数据流
1. Telegram 指令进入对应 Bot（PM Bot 或 DEV+QA Bot）。
2. Bot 转发指令到对应 OpenClaw 实例。
3. 实例执行任务并回传结果到 Bot，再回传给用户。
4. 跨角色协作信息通过外部共享载体（如共享文档/工单/知识库）同步，避免运行态耦合。
5. 群聊策略：允许加入同一群聊，但 Bot 仅处理命令或被@消息，避免误触发并保持可控性。

## 错误处理
- 实例级故障：单实例崩溃仅影响对应角色；守护进程自动重启并告警。
- 通讯故障：Telegram 通道短暂不可用时保留任务记录与日志，恢复后补发/重试。
- 资源异常：为 PM 与 DEV+QA 实例设置 CPU/内存上限，避免互相拖垮。
- 权限误投：多 Bot 隔离作为默认防线；若改为单 Bot 路由，必须强制角色标签与白名单校验。
- 群聊控制：默认仅响应命令/被@消息，降低噪音与误操作风险。

## 测试
- 连通性：两套 Bot 分别发送指令，确认可达对应实例并返回结果。
- 隔离验证：停掉 PM 实例时 DEV+QA 仍正常，反之亦然。
- 资源限制：压测 DEV+QA 任务，确认 PM 实例性能不受影响。
- 恢复验证：模拟 Telegram 通道中断，恢复后任务可重试/补发。

## 约束与说明
- 实例独立进程/容器，端口/配置/运行目录不得冲突。
- 不共享锁或强依赖，确保单实例故障不影响其它实例。
